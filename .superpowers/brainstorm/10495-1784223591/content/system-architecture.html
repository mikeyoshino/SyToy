Section 1 — System Architecture & Business Boundaries

หนึ่ง modular monolith แบ่งตาม business capability และใช้ PostgreSQL ฐานเดียว

Identity

CustomerAdmin
  • Email/password
  • Saved addresses ≤ 5
  • Policy authorization

Catalog

  • Product — no variants
  • ArtToy / Gundam
  • Brand, Universe
  • Many Characters
  • Ordered images

Inventory & Checkout

  • Stock movements
  • CheckoutAttempt
  • 30-min reservation
  • 2-min safety grace
  • Concurrency control

Payments & Orders

  • Stripe Embedded
  • Verified webhooks
  • Order after payment
  • Immutable snapshots
  • Deposit + balance

Fulfillment & Ops

  • Shipment + tracking
  • Email deliveries
  • LINE Messaging API
  • Sales dashboard
  • Retry/audit records

Customer purchase flows

IN-STOCK
Browse → Anonymous cart → Login & merge → CheckoutAttempt + stock reservation → Stripe Embedded → Verified webhook → Order: Paid + ReadyToShip
PRE-ORDER
Product detail → Login → Direct pre-order checkout → Capacity reservation → Stripe deposit → Order: DepositPaid + AwaitingArrival → Admin marks arrival → 7-day balance request → Stripe balance payment → Paid + ReadyToShip
Critical boundary: ก่อนชำระยังไม่มี Order แต่มี CheckoutAttempt + Reservation ที่ durable; Order จริงเกิดจาก verified payment เท่านั้น และ PaymentStatus แยกจาก FulfillmentStatus
